# 공모전 서면·PT 공통 서사 (NextSpot)

> 2026 관광데이터 활용 공모전(웹 개발부문) 서면심사·PT심사에서 공통으로 쓰는 서사 원고.
> `docs/CONTEST_STRATEGY.md` C1(독창성 명문화)·C2(시의성)·C3(페르소나 여정)·D6(수익모델)의 실행 산출물이다.
>
> **작성 원칙**: 모든 문장은 실제 코드/스키마에 근거한다. 아직 구현되지 않은 기능(TourAPI 실적재 등)은
> "구현됨"으로 서술하지 않고 현재 상태 그대로 표기한다. 수치가 실측이 아닌 시나리오 예시인 경우
> "예시" 또는 "가정"으로 명시하고, 보도·통계 인용이 필요한 대목은 "보도 인용 필요"로 표기한다.

## 1. 독창성 5종 (C1)

NextSpot은 "혼잡한 곳을 보여준다"가 아니라 "지금 출발하면 도착했을 때 여유로운 곳을 골라준다"는
한 문장으로 요약되는 재배치 엔진이다. 그 한 문장을 구성하는 다섯 가지 독창 요소는 아래와 같다.

### ① 도착시점 예측 (현재 혼잡이 아니라 '내가 도착할 시각'의 혼잡을 예측)

경쟁 서비스 다수는 "지금" 혼잡도를 보여주는 데 그친다. NextSpot의 SPOT 산식은 사용자의 이동 시간을
먼저 계산한 뒤, `현재시각 + 이동시간`으로 산출한 **도착 예상 시점**의 `hour`·`dow`를 예측 모델에 넣어
그 시점의 혼잡도를 추론한다. 대기시간(w2)도 동일한 도착시점 기준으로 계산해, "예측은 미래 시점인데
대기시간은 현재 시점"이라는 시점 불일치를 제거했다.

> 근거: `apps/api/app/services/spot/score.py:69-92` (`arrival_dt = datetime.now(timezone.utc) + timedelta(minutes=travel_time_min)` → `predict_congestion(type, arrival_hour, arrival_dow)` → `calculate_predicted_wait_time(..., hour=arrival_hour)`)

### ② w3 결합형 인센티브 (개인 효용 + 공익 재배치 기여 + 소상공인 제휴를 한 산식에)

SPOT 스코어는 `w1·선호유사도 − w2·시간비용 + w3·인센티브`로 구성된다. 이 중 w3(인센티브) 항은 서로
다른 두 이해관계자의 이익을 하나의 연속값으로 결합한 것이 독창점이다.

- **쿠폰강도**: 제휴 소상공인의 할인율(`coupon_rate`)을 20% 캡으로 정규화한 연속값 — 소상공인 이익.
- **재배치기여**: `원본 혼잡도 − 후보의 도착시점 예측 혼잡도`(0~1) — 사용자를 혼잡지에서 여유지로
  옮길수록 커지는 값으로, 개인 추천이면서 동시에 B2G 관점의 수요 분산 지표가 된다 — 공익.

두 성분을 `INCENTIVE_COUPON_SHARE=0.5`로 절반씩 결합해 "개인이 받는 혜택"과 "사회가 얻는 분산 효과"를
분리된 지표가 아니라 추천 순위 자체에 내재화했다.

> 근거: `apps/api/app/services/spot/score.py:97-104` (`incentive = INCENTIVE_COUPON_SHARE * coupon_term + (1 - INCENTIVE_COUPON_SHARE) * relief_term`), 컬럼 정의: `supabase/migrations/20260707150000_add_coupon_incentive.sql`

### ③ 음성 AI 비서

지도를 손으로 조작하기 어려운 도보 관광 맥락을 고려해, 발화 한 턴을 의도(수락/다음/거절/상세/선호필터/중지)로
분류하고 한국어 응답을 생성하는 음성 인터페이스를 백엔드 엔드포인트로 제공한다. "고깃집" 같은 모호한
발화는 백엔드가 구체 검색어로 확장한 뒤 임베딩 의미검색으로 후보를 재필터링해, 다른 업종 후보가
섞여 나오는 것을 막는다.

> 근거: `apps/api/app/routers/recommendations.py:416-464` (`POST /api/v1/voice/turn`), 프런트 공용 훅 `apps/web/lib/useVoiceAssistant.ts`

### ④ B2G 관제 → 개입 폐루프 (쿠폰 정책이 실시간으로 추천 순위에 반영)

지자체/관광공사 운영자는 관리자 API로 특정 시설의 제휴 할인율(`coupon_rate`)을 즉시 변경할 수 있고,
그 변경은 캐시나 재학습 없이 **다음 추천 요청부터 바로** w3 쿠폰강도 계산에 반영된다. 즉 "정책 개입 →
추천 결과 변화"가 배치 주기 없이 닫힌 루프로 동작한다. 그 개입의 효과는 `/admin/impact`가 수락된
추천의 `원본 예상대기 − 대안 예상대기` 합산으로 정량 집계해 되돌려준다.

> 근거: 정책 개입 `apps/api/app/routers/admin.py:39-44`(`FacilityUpdate.coupon_rate` PATCH, 주석 "변경 즉시 다음 추천 요청의 w3 쿠폰강도에 반영") · 효과 측정 `apps/api/app/routers/admin.py:239-291`(`GET /api/v1/admin/impact`)

### ⑤ 피드백 학습 선호 벡터

사용자가 추천을 수락/거절할 때마다 8차원 선호 벡터가 시설 벡터 방향으로 소폭(수락 +10%, 거절 −5%)
이동하며 갱신된다. 다음 추천 요청부터는 갱신된 벡터가 코사인 유사도(w1)에 즉시 반영되어, 별도의
오프라인 재학습 배치 없이 개인화가 누적된다.

> 근거: `apps/api/app/services/preference_vector_service.py:82-107` (`adjust_user_vector_on_feedback`), 호출부 `apps/api/app/routers/recommendations.py:511-526` (`POST /api/v1/feedback`)

## 2. 시의성 (C2)

- **경주 APEC 이후 관광 수요**: 2025년 11월 APEC 정상회의가 경주에서 개최되며 도시 인지도와 관광
  인프라 투자가 확대된 것은 공지된 사실이나, 그로 인한 후속 방문객 증가폭·시기별 수치는 본 문서에서
  단정하지 않는다 — **PT 발표 시 관광공사·지자체 공식 통계 또는 언론 보도 인용이 필요**하다.
- **황리단길 오버투어리즘**: 첨성대·대릉원·황리단길 일대가 주말·성수기에 보행자 혼잡과 대기시간
  증가를 겪는다는 것은 다수 언론에서 반복적으로 다뤄진 현상이나, 구체적 대기시간·방문객 수 등의
  정량 수치는 실측 데이터 또는 **보도 인용이 필요**하다.
- NextSpot이 대응하는 지점은 정확히 이 두 사실의 교집합이다 — "APEC 이후 늘어난 방문 수요"가
  "이미 포화 상태인 황리단길 핵심 동선"에 그대로 유입되는 것을 막고, 반경 150m 내 대안 시설로
  분산시키는 것이 서비스의 존재 이유다.

## 3. 페르소나 여정 (C3)

> 아래 시나리오의 인원 수·대기시간·소요시간은 PT 데모 설명을 위한 **예시**이며 실측값이 아니다.
> 각 단계는 실제 앱의 화면 흐름과 호출되는 API를 그대로 따른다.

**페르소나**: 가족 관광객 3인, 첨성대 인근에서 도보 여행 중.

| 단계 | 화면/행동 | 실제 구현 |
|---|---|---|
| 1. 지도 확인 | 메인 지도(`app/main`)에서 첨성대 마커 확인 | Kakao Maps 지도에 시설 마커, 혼잡도별 색상(파랑/초록/노랑/주황) 표시 |
| 2. 혼잡 확인 | 첨성대 마커 클릭 → "실시간 혼잡도: 혼잡 · 수용현황 N/M명" 카드 | `congestion_logs` 최신 행 기반, 합성값 없이 로그 부재 시 "데이터 없음" 표기 |
| 3. 대안 추천 | "대안 찾기" → `/explore/recommend` 이동 | `POST /api/v1/recommendations` — 반경 150m 후보에 SPOT 스코어(도착시점 예측 포함) 산정, 쿠폰 있는 한적한 곳이 상위 랭크될 수 있음(w3) |
| 4. 수락 (예시: 3분 내) | 추천 카드에서 "수락" 탭 또는 음성으로 "여기로 가자" | `handleAccept` → `POST /api/v1/feedback`(`action=accepted`) → `recommendations.accepted=true` 갱신 + 선호 벡터 학습 |
| 5. 길안내 | 수락 즉시 카카오맵으로 경로 오픈 | 모바일은 `kakaomap://route` 딥링크, PC/웹은 `map.kakao.com` 길찾기 URL로 이동(`apps/web/app/explore/recommend/page.tsx`) |
| 6. 학습 반영 | 다음 추천부터 벡터 반영 | 수락한 시설 벡터 방향으로 선호 벡터가 이동해, 이후 추천의 w1(선호유사도) 계산에 즉시 반영 |

이 여정에서 "쿠폰 있는 한적한 곳"이 우선 노출되는 이유는 임의 배치가 아니라 §1-②의 w3 산식 —
할인율이 높을수록, 그리고 도착시점 기준으로 더 여유로워질수록 점수가 올라가는 구조 — 때문이다.

## 4. 수익모델 (D6)

**B2G 라이선스 (관제 대시보드)**
지자체·관광공사에 관제 대시보드(`/admin/*`, 공유 토큰 인증 `require_admin`)를 SaaS 라이선스로
제공한다. 시설 CRUD, 쿠폰 정책 개입, 수락률·DAU 지표(`/admin/metrics`), 분산 효과 집계
(`/admin/impact`)가 이미 코드로 구현된 별도 관리자 API 표면이므로 별 도메인 라이선스 판매로
포장하기에 추가 개발 부담이 적다.

**소상공인 제휴 티어 (coupon_rate 연속값 상생 구조)**
`facilities.coupon_rate`는 0/1이 아닌 연속값이며, 할인율이 높을수록 w3 쿠폰강도가 커져 추천 노출
순위가 올라간다(`score.py`). 이는 "제휴 등급이 높을수록 더 많이 노출된다"는 광고 상품 구조가 이미
산식에 내장돼 있다는 뜻이며, 이 등급을 유료 제휴 티어(예: 기본/실버/골드 할인율 구간)로 상품화할 수
있다.

**데이터 리포트 판매**
`/admin/metrics`·`/admin/impact`는 실제 `recommendations`/`user_feedback` 테이블을 집계해 수락률·
재배치 건수·절감 대기시간을 산출한다. 현재는 대시보드 열람용으로만 제공되지만, 동일 집계를 주기적
리포트(PDF/API)로 지자체·상권에 판매하는 것은 별도 판매 기능 개발 없이 집계 로직을 재사용해 확장
가능한 방향이다. (판매 기능 자체는 아직 코드에 없음 — 확장 여지로 서술)
